# 설계 결정 (DECISIONS)

이 문서는 과제 구현 중 내린 결정과 그 이유를 정리합니다. 평가 기준이 "무엇을 했는지"보다 "왜 그렇게 했는지"를 본다고 되어 있어, 구현 내용 자체보다 선택한 이유와 포기한 부분을 중심으로 작성했습니다.

## 1. 상태 구조

서버 상태는 mock API가 가지고 있는 데이터를 기준으로 두고, 클라이언트에서는 `Board` 컴포넌트의 `tasks` state로 현재 화면에 보여줄 태스크 목록을 관리했습니다.

초기 데이터는 `getTasks`로 서버에서 가져오고, 생성/수정/삭제/이동이 성공하면 서버 응답으로 로컬 태스크를 갱신했습니다. 특히 `version`, `updatedAt` 같은 값은 클라이언트에서 임의로 맞추기보다 서버 응답을 기준으로 반영하는 것이 안전하다고 판단했습니다.

상태를 `Board`에서 관리한 이유는 현재 화면이 단일 보드 중심이고, 태스크 목록을 여러 페이지나 전역 영역에서 공유하지 않기 때문입니다. `Column`과 `Card`는 렌더링과 사용자 입력을 담당하고, 실제 태스크 목록 변경과 API 호출은 `Board`에서 처리하도록 역할을 나눴습니다.

TanStack Query 도입도 고려했습니다. 서버 상태 캐싱, refetch, retry, mutation 상태 관리에는 장점이 있지만, 이번 과제에서는 낙관적 업데이트, rollback, 빠른 연속 이동 race condition을 직접 구현하고 설명하는 것이 더 중요하다고 판단했습니다. 또한 현재 화면은 단일 보드 중심이라 React state만으로도 상태 흐름을 명확히 관리할 수 있다고 봤습니다.

## 2. 낙관적 업데이트 & 롤백

이 과제의 mock API는 지연과 실패가 있기 때문에, 서버 응답을 기다린 뒤에만 화면을 바꾸면 사용자가 조작이 느리게 느껴질 수 있다고 판단했습니다. 그래서 이동, 수정, 삭제, 생성 모두 UI를 먼저 반영하도록 구현했습니다.

작업별 처리 방식은 다음과 같습니다.

- 이동: 카드를 먼저 새 컬럼으로 옮기고, `updateTask`를 호출했습니다.
- 생성: 임시 id를 가진 태스크를 먼저 목록에 추가하고, `createTask`를 호출했습니다.
- 수정: 수정 내용을 먼저 카드에 반영하고, `updateTask`를 호출했습니다.
- 삭제: 확인 후 카드를 먼저 제거하고, `deleteTask`를 호출했습니다.

요청이 성공하면 낙관적 업데이트 상태를 그대로 끝내지 않고, 서버 응답으로 해당 태스크를 다시 교체했습니다. `version`, `updatedAt` 같은 값은 서버가 최종 기준이기 때문에 서버 응답을 반영하는 것이 안전하다고 판단했습니다.

요청이 실패하면 이전 상태로 rollback했습니다. 이를 위해 각 작업을 실행하기 전에 현재 `tasks`를 이전 스냅샷으로 보관했고, 실패 시 `setTasks(previousTasks)` 또는 임시 태스크 제거 방식으로 되돌렸습니다.

- 이동 실패: 이동 전 목록으로 복원
- 생성 실패: 먼저 추가했던 임시 태스크 제거
- 수정 실패: 수정 전 목록으로 복원
- 삭제 실패: 삭제 전 목록으로 복원

롤백은 서버 재조회가 아니라 이전 스냅샷 복원을 선택했습니다. 실패 직후 서버를 다시 조회하면 네트워크 상태가 불안정한 상황에서 다시 실패할 수 있고, 5,000개 목록을 매번 다시 가져오는 비용도 큽니다. 사용자가 조작하기 전의 상태로 정확히 되돌리는 데는 이전 스냅샷 복원이 더 단순하고 명확하다고 판단했습니다.


## 3. 경쟁 상태(race condition)

최종 상태가 어긋나지 않도록 하기 위해 카드별 최신 이동 요청 순번을 `useRef`에 저장했습니다. 카드를 이동할 때마다 해당 카드의 요청 순번을 증가시키고, 응답이 돌아왔을 때 요청 당시 순번과 현재 최신 순번이 같은지 비교했습니다.

순번이 다르면 이미 더 최신 이동 요청이 있었다는 뜻이므로, 해당 응답은 성공이든 실패든 화면에 반영하지 않고 무시했습니다. 그래서 최종적으로는 사용자가 마지막으로 이동한 상태가 우선됩니다. 이 방식은 요청 순번 비교를 이용한 last-write-wins 정책입니다.

요청 취소 방식도 고려할 수 있지만, 이미 서버에 도착한 요청은 취소가 보장되지 않습니다. 그래서 요청 자체를 취소하기보다, 응답이 돌아왔을 때 최신 요청인지 확인하고 반영 여부를 결정하는 방식을 선택했습니다.

`version`은 서버와 클라이언트가 같은 데이터 버전을 기준으로 수정하는지 확인하기 위한 값이고, `useRef`의 요청 순번은 브라우저 안에서 오래된 응답을 무시하기 위한 값입니다. 두 값의 역할이 다르기 때문에 함께 사용했습니다.

## 4. 대량 데이터 성능 (5,000개)

5,000개 태스크를 모두 DOM에 렌더링하면 스크롤과 드래그가 느려질 수 있다고 판단해 `@tanstack/react-virtual`을 도입했습니다. 각 컬럼은 전체 태스크를 모두 렌더링하지 않고, 현재 화면에 보이는 카드와 overscan 범위의 카드만 렌더링하도록 했습니다.

검색과 우선순위 필터는 서버 요청 없이 클라이언트에서 처리했습니다. 전체 태스크 목록은 이미 `getTasks`로 한 번 로드되어 있고, 제목 검색과 우선순위 필터는 단순 배열 필터링으로 충분하다고 판단했습니다.

필터링 흐름은 `tasks -> filterByTitle -> filterByPriority -> byStatus -> Column -> virtualized Card rendering` 순서입니다. 이렇게 구성하면 검색/필터 후에도 실제 DOM 렌더링 수가 전체 5,000개에 비례하지 않고, 현재 화면에 필요한 카드 수에만 비례합니다.

인라인 수정 폼처럼 카드 높이가 바뀌는 경우에는 처음에 카드가 겹치는 문제가 있었습니다. 이를 해결하기 위해 `measureElement`와 `data-index`를 사용해 실제 row 높이를 다시 측정하도록 했습니다.

정밀한 Performance 프로파일링까지는 하지 못했습니다. 대신 브라우저에서 5,000개 데이터를 기준으로 스크롤, 드래그, 검색, 우선순위 필터, 수정 폼 열기 동작을 수동으로 확인했습니다. 또한 콘솔에서 `document.querySelectorAll('.card').length`를 확인해, 전체 태스크 수와 실제 렌더링된 카드 DOM 수가 같지 않은 것을 확인했습니다. 이를 통해 가상화가 전체 5,000개 카드를 한 번에 렌더링하지 않고, 현재 viewport와 overscan 범위의 카드만 렌더링하는 구조로 동작하는 것을 확인했습니다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

### 네트워크 완전 단절 시 동작

네트워크가 완전히 끊긴 경우에는 변경을 로컬에 계속 쌓아두기보다, 저장 실패를 명확히 알리고 이전 상태로 되돌리는 정책이 더 적절하다고 생각했습니다.

오프라인 큐를 만들어 사용자의 변경을 임시 저장한 뒤 나중에 동기화하는 방법도 가능하지만, 생성/수정/삭제/이동 요청의 순서와 충돌을 함께 관리해야 합니다. 특히 WMS 같은 운영 화면에서는 저장되지 않은 상태가 저장된 것처럼 오래 보이면 사용자가 잘못된 업무 상태를 기준으로 판단할 수 있다고 생각했습니다.

그래서 이번 구현에서는 초기 조회가 실패하면 에러 화면과 재시도 버튼을 보여주고, 쓰기 요청이 실패하면 낙관적 업데이트로 바뀐 UI를 이전 스냅샷으로 rollback한 뒤 toast로 실패 사실을 알리도록 했습니다. 이 방식은 오프라인 상태에서 계속 작업할 수는 없지만, 서버 상태와 클라이언트 화면이 어긋난 상태로 오래 남는 것을 줄일 수 있다고 판단했습니다.

### 409 충돌 시 사용자에게 보여줄 내용과 해소 방식

409 충돌은 500 서버 오류와 의미가 다르다고 판단했습니다. 500은 서버가 요청을 처리하지 못한 상황이고, 409는 클라이언트가 오래된 `version`을 기준으로 변경을 시도했지만 서버에는 이미 더 최신 변경이 있는 상황입니다.

그래서 409가 발생하면 일반 서버 오류 메시지가 아니라 "다른 변경사항이 먼저 저장되어 현재 변경을 적용하지 못했습니다. 변경을 되돌렸습니다."라는 별도 toast를 보여주도록 했습니다. 사용자가 단순한 서버 실패가 아니라 데이터 충돌 때문에 변경이 반영되지 않았다는 점을 알 수 있게 하기 위해서입니다.

해소 방식은 자동 병합이나 자동 덮어쓰기가 아니라 rollback을 선택했습니다. 409 상황에서 클라이언트 변경을 자동으로 다시 덮어쓰면, 서버에 먼저 저장된 다른 변경을 잃을 수 있다고 생각했습니다. 특히 운영 화면에서는 누가 더 최신 상태를 만들었는지 모르는 상태에서 자동으로 덮어쓰는 것이 위험하다고 생각했습니다.

따라서 이번 구현에서는 낙관적 업데이트로 바뀐 화면을 이전 스냅샷으로 되돌리고, 사용자가 최신 상태를 기준으로 다시 시도하도록 했습니다. 완전한 충돌 해결 UI, 예를 들어 서버 최신값과 내 변경값을 비교해서 선택하는 화면은 구현하지 못했지만, 최소한 409와 500을 구분해 안내하고 저장되지 않은 상태가 화면에 남지 않도록 했습니다.

### 실패한 쓰기의 재시도: 자동 vs 수동

실패한 쓰기 요청은 자동 재시도보다 사용자가 직접 다시 시도하는 수동 방식을 선택했습니다.

이번 과제는 느리고 가끔 실패하는 서버 위에서 보드가 견고하게 동작해야 하는 상황입니다. 즉, 쓰기 요청 실패가 단순히 아주 드문 예외라기보다 실제로 발생할 수 있는 조건이라고 생각했습니다. 이런 상황에서 자동으로 쓰기 요청을 다시 보내면 사용자가 이미 취소하고 싶은 변경이 나중에 반영되거나, 이동/수정/삭제 요청의 순서가 더 복잡하게 꼬일 수 있다고 판단했습니다.

그래서 이번 구현에서는 쓰기 요청이 실패하면 먼저 낙관적 업데이트로 바뀐 화면을 rollback하고, toast로 실패 사실을 알려 사용자가 직접 다시 시도하게 했습니다. 이 방식은 자동 재시도보다 편리함은 떨어지지만, 사용자가 현재 변경이 저장되지 않았다는 사실을 명확히 알 수 있고 화면 상태도 예측하기 쉽다고 생각했습니다.

## 6. 그 외 트레이드오프 / 남은 한계

제한된 시간 안에서는 Priority 1 요구사항과 사용자가 직접 체감할 수 있는 안정성 개선을 우선했습니다. 그래서 5,000개 렌더링 성능, 실패 시 rollback, 빠른 연속 이동 race condition, 생성/수정/삭제, 검색/우선순위 필터를 먼저 구현했습니다.

포기한 부분은 다음과 같습니다.

- 409 충돌 시 서버 최신값과 내 변경값을 비교해서 선택하는 충돌 해결 UI
- 실패한 쓰기 요청의 자동 재시도와 backoff
- 여러 탭을 동시에 열었을 때 상태를 맞추는 다중 탭 동기화
- 키보드만으로 카드를 이동하는 접근성 기능
- 검색 debounce, 상태/태그 다중 필터
- 정량적인 Performance before/after 프로파일링

현재 구현은 409와 500을 구분해 toast로 안내하고 rollback하지만, 충돌을 사용자가 직접 비교해서 병합하는 단계까지는 구현하지 못했습니다. 더 시간이 있었다면 409 응답의 서버 최신 데이터를 화면에 보여주고, 사용자가 다시 적용할지 선택하는 충돌 해결 흐름을 먼저 보강했을 것 같습니다.
